# UI 系统分析与自适应重构计划

## 目标

当前 Godot 宿主层 UI 已经完成了主要功能接线，但布局体系仍以 `1920x1080` 固定设计稿坐标为主，局部混用 Godot 容器布局。后续重构目标不是简单修补单个错位图标，而是建立一套稳定、可验证、可逐步迁移的 UI 布局原则，使 HUD、地图、战斗、剧情与各类面板在常见分辨率和宽高比下表现一致。

本计划只覆盖 Godot 宿主层 UI 表现与布局，不改变应用层服务、领域模型、存档结构和剧情命令语义。

## 当前结构判断

### 总体分层

当前 UI 入口集中在 `autoload/ui_root.tscn` 与 `src/Game.Godot/UI/UIRoot.cs`。

现有分层是合理的：

- `HudLayer`：常驻 HUD。
- `PanelLayer`：背包、队伍、角色、日志、英雄、系统、商店、储物箱等主面板与弹出面板。
- `ModalLayer`：剧情对白、剧情选择、确认框、出战选择、战斗界面、开局问卷等模态流程。
- `OverlayLayer`：toast、hint 等覆盖型提示。

这个结构可以保留。后续重构重点不应从 `UIRoot` 大拆开始，而应在每个层内引入统一设计画布和安全区域规则。

### 当前布局类型

当前 UI 混合了三种布局风格：

1. 固定设计稿坐标
   - 大量节点使用 `layout_mode = 0`、`offset_left`、`offset_top`、`scale`。
   - HUD、地图、战斗、角色面板、基础面板中最常见。

2. 局部容器布局
   - 清单区域和重复条目较多使用 `VBoxContainer`、`HBoxContainer`、`GridContainer`、`ScrollContainer`。
   - 背包、商店、储物箱、技能列表、队伍卡片已有可复用基础。

3. 少量设计画布缩放
   - `BattleLegendOverlay` 已经有 `DesignSize = 1920x1080` 和 resize 后等比缩放的思路。
   - 这应作为后续通用方案的雏形，而不是只保留在战斗大招 overlay 内。

### 错位主要来源

项目显示设置为：

```text
viewport_width = 1920
viewport_height = 1080
stretch/mode = canvas_items
stretch/aspect = expand
```

`aspect = expand` 会在非 `16:9` 宽高比下扩展画布。固定坐标节点不会自动贴合新边界，因此会出现以下问题：

- 右下 HUD 按钮组使用固定 `offset_left = 1730`、`offset_top = 885`，不会真正贴右下角。
- 顶栏货币、时间、地图名按固定坐标分布，宽高比变化后不会重新分配空间。
- toast 背景和文本固定在全屏坐标，宽屏或窄屏时不一定保持视觉居中。
- 地图背景会 stretch，但地图点位和主角 pin 使用固定比例计算，可能与图像实际显示区域不一致。
- 战斗 UI 的右上按钮、底部行动栏、棋盘位置多数是固定坐标，窗口比例变化后容易偏移或压边。
- `JyPanel` 基础面板的背景和关闭按钮使用固定坐标，所有继承面板都会继承这个风险。

## 原版设计思路判断

现有资源和场景摆法更接近传统单机 RPG 的固定画布 UI：

- 使用大量完整底图、按钮贴图、头像框、装饰框。
- 视觉位置来自美术稿坐标，而不是响应式布局。
- 交互区域围绕固定视觉资产摆放。
- 主要目标是还原原版视觉，而不是适配任意屏幕。

因此后续不要把它改成普通 Web/SaaS 那种完全流式布局。更合适的专业方案是：

- 背景层允许铺满、裁切、扩展。
- 交互 UI 层使用稳定的 `1920x1080` 设计安全画布。
- 贴边控件在安全画布内使用锚点和 margin。
- 列表和文本内容区域使用 Godot 容器系统。

## 重构原则

### 小步迭代与手动提交

后续 UI 布局迁移按小步闭环推进，尤其是剧情、开局流程、地图和战斗这几类高风险区域。

- 每次只迁一个小目标，优先选择可以独立打开、独立验证的场景或坐标模型。
- 每次完成后停止继续迁移，输出改动范围、静态检查结果、手动验证清单和建议 Conventional Commit 提示词。
- Codex 不自动 stage、不自动 commit；提交由用户在手动验证通过后执行。
- 用户确认继续后，再进入下一轮。
- 如果某一步发现坐标模型不清晰，先补模型和文档，再迁依赖它的节点。

### 统一设计画布

引入统一的 `DesignCanvas` 概念：

- 设计尺寸固定为 `1920x1080`。
- 根据实际 viewport 使用等比缩放。
- 默认使用 `min(viewportWidth / 1920, viewportHeight / 1080)`。
- 设计画布居中。
- 画布外的额外空间只允许背景层或特殊全屏效果使用。

这样可以保留原版固定美术坐标，又能避免 UI 在宽屏或高屏下漂移。

### 背景与交互分离

每个全屏界面应分成两层：

- `BackgroundRoot`
  - 负责地图背景、战斗背景、主菜单背景、商店背景等。
  - 可以全屏 expand/crop。

- `UiRoot` 或 `DesignRoot`
  - 负责按钮、文本、面板、可点击区域。
  - 挂在 `DesignCanvas` 下。
  - 使用统一设计坐标和安全区域。

地图和战斗尤其需要这条规则。背景怎么拉伸，与交互点位怎么换算，必须由同一个坐标模型明确处理。

### 贴边控件用锚点

所有屏幕边缘控件不得再依赖硬编码接近边缘的设计坐标，例如：

- 右下按钮组。
- 右上系统/战斗按钮。
- 顶部 HUD。
- 底部行动栏。
- toast 和 hint。

这类控件应表达为：

- 锚到 `top-left`、`top-right`、`bottom-left`、`bottom-right`、`center`。
- 使用统一 margin。
- 尺寸由设计画布或容器决定。

### 内容区域用容器

重复项、列表、按钮组、文本区应优先使用 Godot 容器：

- `MarginContainer`
- `PanelContainer`
- `VBoxContainer`
- `HBoxContainer`
- `GridContainer`
- `ScrollContainer`

固定坐标只允许用于装饰图、整张美术框和必须按美术稿对齐的局部视觉元素。即便使用固定坐标，也应位于 `DesignCanvas` 内，而不是直接放在全屏根节点下。

### 保留现有业务接线

本次重构不改变：

- `GameSession`
- `UIRoot` 对外 API
- 应用层服务调用
- 剧情命令语义
- 战斗内核
- 存档结构

UI 脚本可以逐步整理，但每一步都应先保证行为不变。

### 不做兼容性补丁堆叠

不要为每个分辨率写一套特殊偏移，也不要在脚本里到处加 `if width > xxx`。正确方向是统一坐标系统和布局规则。

临时修错可以接受，但必须服务于最终结构。例如先把 HUD 右下按钮迁到 `DesignCanvas`，而不是只把 `1730` 改成另一个魔法数字。

## 详细分析计划

### 1. 建立 UI 清单

按层级列出所有场景：

- 常驻：`hud`
- 地图：`map_screen`、`map_entity_slot`、`map_entity_box`
- 剧情：`story_dialogue_panel`、`story_choice_panel`、`hint_box`
- 基础：`jy_panel`、`jy_button`、`confirm_dialog`
- 主面板：角色、队伍、背包、商店、储物箱、日志、英雄、系统
- 开局流程：门派选择、输入名字、选择头像、随机属性
- 战斗：出战选择、战斗主界面、棋盘、技能栏、物品面板、结算面板、飘字、大招 overlay
- 主菜单与失败/通关界面

为每个场景标注：

- 是否全屏。
- 是否依赖 `JyPanel`。
- 是否有贴边控件。
- 是否使用固定 offset。
- 是否使用容器。
- 是否有动态生成子节点。
- 是否与世界背景、地图坐标、战斗棋盘坐标耦合。

### 2. 标注错位风险

按风险分级：

- P0：影响点击或核心流程，例如 HUD 入口、战斗按钮、地图点位。
- P1：视觉明显错位，但不阻断流程，例如 toast、底部描述框、面板关闭按钮。
- P2：局部不美观，例如列表间距、文本溢出。

优先处理 P0。

### 3. 选定验证分辨率

后续每次迁移至少验证：

- `1920x1080`：设计基准。
- `1280x720`：等比缩小。
- `2560x1440`：等比放大。
- `3440x1440`：超宽屏。
- `1920x1200`：16:10。
- `1366x768`：常见低分辨率。

需要记录：

- 是否裁切。
- 是否贴边正确。
- 是否按钮可点击。
- 是否文本溢出。
- 是否面板遮挡关键区域。
- 是否地图点位与背景一致。
- 是否战斗棋盘、单位、飘字对齐。

### 4. 明确每类 UI 的目标形态

不同 UI 不应套同一模板：

- HUD：安全画布内贴边布局。
- 主面板：居中设计画布 + 容器化内容区。
- 地图：背景全屏，地图交互层使用与地图图像一致的坐标转换。
- 战斗：背景全屏，棋盘和行动 UI 在设计画布内稳定定位。
- 剧情：底部对白框与选项面板锚到底部安全区。
- Toast/Hint：锚到顶部或中心安全区，不依赖绝对屏幕坐标。

## 重构计划

### 阶段 0：建立验证基线

目标：在不改 UI 行为的情况下，建立测试和截图基线。

工作：

- 记录当前 `1920x1080` 下的 HUD、地图、剧情、背包、角色、商店、战斗截图。
- 记录至少一个超宽屏和一个 16:10 下的错位截图。
- 整理 P0/P1/P2 错位清单。

验收：

- 有明确的“改前”对照。
- 后续每次改动能确认是修复还是引入新偏差。

### 阶段 1：抽通用设计画布

目标：提供所有后续 UI 迁移的基础设施。

建议新增：

- `DesignCanvas` 控件脚本。
- 统一常量 `DesignSize = 1920x1080`。
- 可选 `SafeAreaMargin`。
- resize 时自动计算 scale、position、size。

设计要点：

- `DesignCanvas` 自身填满 viewport。
- 内部 `DesignRoot` 尺寸固定为 `1920x1080`。
- `DesignRoot` 等比缩放并居中。
- 支持获取设计坐标到屏幕坐标、屏幕坐标到设计坐标。

验收：

- 单独测试空场景中 `DesignRoot` 在多分辨率下居中且等比。
- 不改业务 UI 行为。

### 阶段 2：迁 HUD

目标：优先解决屏幕图标错位。

工作：

- 将 HUD 交互节点放入 `DesignCanvas/DesignRoot`。
- 右下按钮组改为 `bottom-right` 锚点或设计画布内明确 margin。
- 左上主角入口、顶部信息栏、难度图标统一进入设计画布。
- 顶栏可以暂时保留原图和坐标，但必须处于设计画布内。

验收：

- 右下按钮组在 16:9、16:10、超宽屏中位置稳定。
- 按钮点击区域与图标一致。
- HUD 隐藏/显示逻辑不变。
- `RunInfo` tooltip 正常。

### 阶段 3：迁 Overlay 与基础弹窗

目标：修复 toast、hint、confirm、基础 panel 的通用错位。

工作：

- `ToastPanel` 使用顶部或中心安全区锚点。
- `HintBox` 使用安全画布居中或按剧情需求定位。
- `ConfirmDialog` 保持居中，并由容器控制按钮和文本。
- `JyPanel` 拆出背景层、内容根、关闭按钮锚点。

验收：

- toast 在所有测试分辨率下居中或稳定贴上方。
- 确认框居中，不被裁切。
- 所有继承 `JyPanel` 的主面板关闭按钮位置一致。

### 阶段 4：迁主面板

目标：将背包、角色、队伍、日志、英雄、系统、商店、储物箱统一到可维护布局。

当前进度：

- 已完成：`SystemPanel`、`SaveSlotSelectionPanel`、`InventoryPanel`、`ShopPanel`、`ChestPanel`、`PartyPanel`、`JournalPanel`、`HeroPanel`。
- 已完成基础壳：`JyPanel` 的通用背景与关闭按钮已进入设计画布。
- 已完成：`CharacterPanel`、装备选择、物品目标选择、出战选择、战斗物品、战斗结算等剩余 `JyPanel` 子面板。

建议剩余顺序：

1. 剧情对白与选项。
2. 开局流程。
3. 地图 UI。
4. 战斗棋盘和行动栏。

原因：

- 角色面板依赖头像、五个分页按钮、多个 tab，视觉坐标更重，应独立处理。
- 战斗弹窗可以先迁，但战斗棋盘、单位、飘字和技能动画坐标必须留到战斗阶段整体处理。

验收：

- 面板打开、关闭、切 tab、列表滚动正常。
- tooltip 不超出可视区域。
- 动态列表在低分辨率下可滚动，不遮挡按钮。
- 角色面板的装备、技能、天赋切换行为不变。

### 阶段 5：迁剧情 UI 与开局流程

目标：对白、选项、门派选择、改名、头像选择、随机属性适配多分辨率。

工作：

- 第一步只迁剧情对白框。
  - `scenes/ui/story/story_dialogue_panel.tscn`
  - 对白框锚到底部安全区。
  - 长文本滚动、文本区左键继续、空白区左键继续行为保持不变。
- 第二步只迁剧情选项框。
  - `scenes/ui/story/story_choice_panel.tscn`
  - `scenes/ui/story/story_choice_button.tscn`
  - 选项框锚到对白框上方或中心区域。
  - 选项按钮使用容器控制，避免低分辨率溢出。
- 第三步迁开局流程第一批。
  - `scenes/ui/select_sect/select_sect_screen.tscn`
  - `scenes/ui/start_question/input_name_panel.tscn`
  - 门派选择和输入名字进入设计画布。
- 第四步迁开局流程第二批。
  - `scenes/ui/start_question/select_head_panel.tscn`
  - `scenes/ui/start_question/select_head_slot.tscn`
  - `scenes/ui/start_question/roll_stats_panel.tscn`
  - 头像选择和随机属性界面进入设计画布。

验收：

- 长文本可滚动。
- 点击对白区域继续行为不变。
- 选项按钮不溢出。
- 输入名字、选择头像、roll stats 流程不变。
- 每一步独立验收后停止，由用户手动验证并提交。

### 阶段 6：迁地图 UI

目标：解决地图点位、主角 pin、底部描述框、地图按钮在不同分辨率下的稳定性。

工作：

- 第一步只建立地图坐标模型。
  - 背景层继续全屏。
  - 明确地图图片实际显示 rect。
  - 大地图交互层建立明确的地图坐标空间。
  - 地图位置数据、主角 pin、点击命中都必须使用统一 transform。
- 第二步再迁地图覆盖 UI。
  - `scenes/map/map_entity_box.tscn`
  - `scenes/map/map_entity_slot.tscn`
  - 小地图点位列表使用容器或安全画布固定区域。
  - 底部描述框锚到底部安全区。

验收：

- 大地图点位与背景视觉位置一致。
- 主角 pin 与当前地图位置一致。
- 小地图点位列表不溢出。
- 剧情播放时隐藏地图交互点和底部信息区的行为不变。
- 每一步独立验收后停止，由用户手动验证并提交。

### 阶段 7：迁战斗 UI

目标：战斗棋盘、按钮、技能栏、单位、飘字、结算面板在不同分辨率下稳定。

工作：

- 第一步只建立战斗棋盘坐标模型。
  - 战斗背景全屏 expand。
  - 棋盘放入设计画布中的稳定区域。
  - 棋盘格、单位、鼠标 hover、可达区域、飘字、技能动画共享棋盘坐标转换。
- 第二步再迁战斗行动 UI。
  - `scenes/ui/battle/battle_skill_box.tscn`
  - `scenes/ui/battle/battle_skill_view.tscn`
  - `scenes/ui/battle/battle_float_text.tscn`
  - `scenes/ui/battle/battle_legend_overlay.tscn`
  - 右上按钮组使用锚点。
  - 底部行动栏使用安全画布底部区域。
  - 复用或泛化 `BattleLegendOverlay` 的设计画布缩放思路。

验收：

- 棋盘格、单位、鼠标 hover、可达区域显示一致。
- 技能释放位置、飘字位置、技能动画位置一致。
- 移动、物品、休息、技能按钮点击正常。
- 战斗结算面板在多分辨率下不被裁切。
- 每一步独立验收后停止，由用户手动验证并提交。

### 阶段 8：清理与规则固化

目标：减少后续 UI 回退到固定全屏坐标的概率。

工作：

- 文档化 UI 场景模板。
- 约定新 UI 必须进入 `DesignCanvas` 或明确说明为什么不进入。
- 约定贴边控件必须使用锚点和 margin。
- 把常见 panel/title/list/button 结构抽成可复用场景或脚本。
- 清理不再使用的 magic offsets。

验收：

- 新增 UI 有明确模板可抄。
- `rg "offset_left = 17[0-9][0-9]" scenes/ui` 这类明显贴边魔法坐标显著减少。
- 主要 UI 在验证分辨率下通过截图检查。

## 推荐优先级

当前 `DesignCanvas`、HUD、toast、confirm、`JyPanel`、系统面板、存档槽选择、主要面板、剧情对白框、剧情选项框、开局流程和地图 UI 已完成代码侧迁移。后续推荐优先级调整为：

1. 战斗棋盘坐标模型。
2. 战斗行动 UI。
3. 主菜单、失败/通关、角色摘要、hint 等扫尾。

地图和战斗属于高风险阶段，不允许把坐标模型、节点迁移和表现验证压进同一大步。

## 测试策略

### 手动测试清单

每次迁移至少检查：

- 主菜单进入新游戏。
- HUD 按钮可点击。
- 打开/关闭队伍、背包、系统、日志。
- 地图点位点击。
- 剧情对白推进和选项选择。
- 商店打开与关闭。
- 储物箱打开与关闭。
- 战斗出战选择与进入战斗。

### 分辨率截图检查

每个阶段至少截图：

- `1920x1080`
- `3440x1440`
- `1920x1200`
- `1366x768`

优先观察：

- 右下角控件。
- 顶部状态栏。
- 中央弹窗。
- 底部对白框。
- 地图点位。
- 战斗棋盘与底部行动栏。

### 自动化方向

后续可以补一个轻量 Godot headless 验证命令：

- 启动指定场景。
- 设置窗口尺寸。
- 等待一帧或数帧。
- 截图输出到临时目录。
- 检查关键节点 rect 是否在 viewport 内。

自动化不要求第一轮完成，但从 HUD 迁移完成后就可以开始补。

## 风险与处理方式

### 风险：视觉还原下降

处理：

- 先在 `1920x1080` 验证与原效果一致。
- 再验证其他比例。
- 不在第一轮重做美术，只改变布局归属和坐标系统。

### 风险：地图点位与背景不一致

处理：

- 地图阶段单独处理。
- 明确地图图片的实际显示 rect。
- 所有点位和 pin 都基于该 rect 换算。

### 风险：战斗 UI 连带动画错位

处理：

- 战斗最后迁。
- 棋盘坐标、单位位置、飘字位置、技能动画位置必须共享同一个 transform。
- 不允许只改按钮而不验证棋盘交互。

### 风险：一次改太多难以回归

处理：

- 每轮只迁一个小目标或一个坐标模型。
- 每轮保留明确改动记录、静态检查和功能验收。
- 每轮结束提供建议 commit 提示词，用户手动提交后再继续。
- 行为逻辑不与布局重构混改。

## 完成标准

短期完成标准：

- HUD 图标在常见分辨率下不再错位。
- toast、confirm、基础面板不明显偏移或裁切。
- 新增 `DesignCanvas` 成为后续 UI 迁移入口。

中期完成标准：

- 所有主面板在 `1920x1080`、`1920x1200`、`3440x1440` 下可用。
- 面板关闭按钮、标题、内容区域位置统一。
- 新 UI 不再直接挂满屏固定坐标。

长期完成标准：

- 地图点位、战斗棋盘、剧情 UI 在多分辨率下稳定。
- UI 布局规则文档化并可复用。
- 固定坐标只存在于设计画布内部的装饰性局部，不再直接依赖实际屏幕尺寸。
