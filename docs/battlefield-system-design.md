# 战场系统功能设计

> 本文用于在暂时移除战场相关实现前，保留战场模块的功能边界、运行时语义和后续重构目标。
>
> 本文是设计记录，不要求当前代码继续保持旧类名，也不以兼容旧实现为目标。后续重构应按本文重新拆分职责，而不是给旧 `BattleEngine` 做补丁式扩展。

## 1. 定位

战场系统负责一次战斗进入后，场上单位在格子地图中的行动、移动、技能施放、资源消耗、状态结算、胜负判定和事件输出。

它不负责：

- 角色长期成长
- 存档槽结构
- 世界地图推进
- 剧情事件分支
- Godot 渲染与输入适配
- 内容文件读取与 JSON materialize

战场系统应当只依赖已装配好的 runtime definition、角色实例投影和确定性随机服务。战场内可以持有运行时对象引用；存档层不保存战场运行态对象引用。

## 2. 核心目标

第一版重构后的战场系统应满足：

- 支持半即时行动条推进，而不是固定 IGOUGO 回合。
- 支持格子地图、正交移动、占位、敌方邻接阻碍等基础战棋规则。
- 支持单位分阵营参战，并通过存活阵营判定胜负。
- 支持移动阶段与行动阶段分离。
- 支持战斗命令由外部输入，战场内只校验合法性并推进状态。
- 支持角色持久化实例投影为战斗单位运行态。
- 支持技能实例投影为可执行战斗动作。
- 支持战斗中资源、冷却、Buff、额外行动、死亡拦截等临时状态。
- 支持 deterministic replay：相同初始内容、相同 seed、相同命令序列应得到相同事件序列。
- 支持结构化战斗事件输出，供 UI、日志、回放和测试消费。

## 3. 主要概念

### 3.1 Battle Encounter

战斗入口定义。它回答“这场战斗从哪里来，以及结束后如何回写外部世界”。

后续应独立于纯战场 definition，至少包含：

- 入口 ID
- 触发来源
- 参战队伍选择策略
- 使用哪个战场定义
- 胜利、失败、逃跑、中断后的结果投影

它属于地图/事件系统与战场系统的衔接层，不应塞进战场运行态。

### 3.2 Battlefield Definition

战场静态定义。它回答“这张战场是什么”。

建议包含：

- `Id`
- `Name`
- 宽高或地图引用
- 地形格数据
- 部署点
- 默认胜负条件
- 可选的战场级规则
- 可选的表现资源引用

当前实现中 `BattleScenarioDefinition` 只包含宽高和参与者列表，后续应拆开：

- 战场地图定义
- 遭遇战定义
- 参战者部署定义

### 3.3 Battle Participant

参战者定义。它回答“谁被放进战斗，以什么阵营、什么初始位置进入”。

建议包含：

- 战斗内单位 ID
- 角色引用或模板引用
- 阵营
- 初始格子
- 初始朝向
- 可选 AI 策略
- 可选入场条件

如果参战者来自玩家队伍，应优先使用 `CharacterInstance` 投影；如果来自敌方模板，则由 `CharacterDefinition` 创建临时实例后投影。

### 3.4 Combatant

战斗单位运行态。它是角色实例在单场战斗中的投影，不是持久化角色本体。

建议包含：

- 战斗内 ID
- 角色 ID 与显示名
- 阵营
- 当前格子
- 部署顺序
- 基础属性快照
- 当前生命、内力、怒气
- 行动条值
- 当前移动/行动阶段标记
- 当前冷却
- 当前 Buff
- 当前有效 attachment 投影
- 当前 hook 索引
- 本场战斗内的 hook 使用计数
- 额外行动次数
- 存活状态

不应把 `Combatant` 设计成角色长期状态的写回容器。战斗结束后的成长、掉落、剧情标记、伤亡处理应通过结果投影统一回写。

## 4. 战场运行态

战场运行态应由一个普通对象承载，不应是全局单例。

建议包含：

- 战场尺寸或地图引用
- 战斗单位集合
- 当前 tick
- 当前事件序号
- 当前活动单位 ID
- 当前阶段
- 当前战斗状态
- 确定性随机服务
- 战斗事件历史
- 可选的 pending action / pending selection

当前阶段第一版可以保留：

- `None`
- `Movement`
- `Action`
- `BattleEnded`

后续如需要加入反应、连击、选择目标、动画锁定，应增加显式阶段，而不是用 nullable 字段隐式表示。

## 5. 行动条与轮转

战场采用半即时行动条。

基本规则：

1. 每个存活单位根据速度积累行动条。
2. 行动条达到阈值的单位进入 ready 集合。
3. 从 ready 集合中选择行动条最高者。
4. 行动条相同或接近时，用速度、部署顺序等稳定规则打破平局。
5. 单位完成行动后扣除行动阈值。
6. 若单位获得额外行动，则不进入全局推进，直接重开自己的移动/行动阶段。

第一版稳定排序建议：

1. 行动条高者优先
2. 速度高者优先
3. 部署顺序小者优先

这组规则必须是确定性的，不能依赖 dictionary 遍历顺序。

## 6. 命令模型

战场系统只接受命令，不直接读取 UI 输入。

当前最小命令集：

- `Move`
- `UseSkill`
- `Rest`

后续可扩展：

- `UseItem`
- `Wait`
- `Guard`
- `InteractBattleObject`
- `SelectReaction`
- `Retreat`

命令执行规则：

- 只有当前活动单位可以执行命令。
- 命令必须匹配当前阶段。
- 战场内负责校验路径、资源、冷却、目标、射程、状态限制。
- 命令成功后输出事件并推进状态。
- 命令失败不改变战场状态。

第一版可以让 `UseSkill` 暂时只包含技能 ID，但完整设计中它必须包含目标选择：

- 主目标单位 ID
- 主目标格子
- 路径或朝向
- 可选的影响区域确认

不应让技能执行时再自动猜目标；AI 可以替玩家生成命令，但命令本身必须完整表达选择。

## 7. 格子与移动

第一版战场格子以正交移动为基础。

基础规则：

- 坐标使用整数格。
- 只允许上下左右移动。
- 路径必须逐格相邻。
- 不能走出战场边界。
- 不能进入其他存活单位占据的格子。
- 移动点来自单位当前有效属性。
- 邻接敌方格子时可增加移动成本，表达敌方控制区。

后续地形系统应拆成独立规则：

- 是否可通行
- 移动成本
- 高度或层级
- 地形标签
- 对技能、伤害、命中、视线的影响
- 阵营或单位类型限制

移动计算应返回可解释结果：

- 可到达格子
- 到达每格的最小成本
- 推荐路径或所有等价路径中的稳定选择
- 不可达原因

## 8. 技能与战斗动作

战场执行层应面对“战斗动作”，而不是直接关心技能学习体系。

技能系统负责把角色当前学习态投影为可执行动作集合：

- 外功本体动作
- 外功附属招式
- 已装备内功的附属招式
- 后续特殊技能动作
- 后续物品动作

动作进入战场后应统一表达：

- 动作 ID
- 来源实例
- 消耗
- 冷却
- 施放范围
- 影响范围
- 目标过滤
- 伤害或治疗模型
- Buff 应用
- 表现资源引用
- 附加 hook 或临时规则

当前代码中 `UseSkill` 尚未表达目标，且技能执行逻辑仍是收敛原型。重构时不应延续“执行时无目标”的接口，应直接把目标选择纳入命令与动作定义。

## 9. 伤害、治疗与资源

战场结算应把数值结算与流程副作用分开。

伤害结算建议遵循：

1. 读取动作、攻击者、目标和上下文。
2. 计算基础伤害。
3. 执行 `BeforeDamageResolve`，收集数值贡献。
4. 统一聚合基础加算、基础乘算、最终加算、最终乘算、覆盖、上限。
5. 执行 `BeforeDamageApply`，处理护盾、减伤上限、伤害取消等受控改写。
6. 若将导致死亡，执行 `BeforeDeath`。
7. 应用扣血或死亡取消。
8. 输出伤害、死亡、击杀事件。
9. 执行 `AfterDamageApplied`。

治疗结算后续应拥有独立阶段，不要长期复用伤害阶段硬塞：

- `BeforeHealResolve`
- `AfterHealApplied`

资源系统第一版至少包含：

- 生命
- 内力
- 怒气

资源变化必须输出事件，事件中要能区分资源类型、变化量和来源。

## 10. Attachment 与 Hook

战场系统应继续把装备、外功、内功、特殊技能、天赋、Buff 都视为 attachment source。

来源可以提供：

- 常驻 modifier
- 战斗 hook
- 表现标记

战斗单位初始化或 attachment 变化后，应刷新：

- 属性 modifier
- 授予天赋
- 动画/表现标记
- 按 phase 预索引的 hook 列表

hook 主干仍采用固定阶段：

- `TurnEnd`
- `BeforeDamageResolve`
- `BeforeDamageApply`
- `BeforeDeath`
- `AfterDamageApplied`
- `AfterAction`

后续必须补齐：

- phase 到 condition / operation 的合法矩阵
- 来源归因
- 同 phase 内排序规则
- target / selector 体系
- 来源、动作、伤害、单位标签
- hook 生命周期与缓存失效策略
- trace / explain 数据结构

这些内容已有细节文档，战场重构时应以 `docs/battle-effect-phase-hook-design.md` 为准。

## 11. Buff 与临时状态

Buff 是单场战斗运行态，默认不直接写回角色长期状态。

Buff 需要明确：

- 来源
- 目标
- 剩余持续时间
- 层数
- 是否可叠加
- 最大层数
- 提供的 modifier
- 提供的 hook
- 移除时机

第一版可在单位回合结束时递减持续时间。后续若需要更精细，应显式区分：

- 来源行动结束
- 目标行动结束
- battle tick
- 受到伤害后
- 触发次数耗尽

Buff 增减后必须刷新单位 attachment 投影和 hook 索引。

## 12. 事件、日志与回放

战场状态变化应通过结构化事件对外输出。

事件至少包含：

- 单调递增序号
- 当前 tick
- 事件类型
- 行为者单位 ID
- 目标单位 ID
- 位置
- definition ID
- 数值
- 详情

事件用途：

- UI 播放
- 测试断言
- 自动战斗回放
- 调试日志
- 后续 battle trace

事件不是存档格式。战斗中断恢复如果后续需要支持，应设计单独的 battle snapshot，而不是依赖事件历史反推所有状态。

## 13. AI 与自动战斗

AI 不应嵌入战场引擎。

推荐边界：

- 战场引擎提供当前状态和合法命令。
- AI 根据策略选择一个合法命令。
- 战场引擎执行命令。

第一版 `BasicBattleAgent` 只能作为测试和占位：

- 移动时靠近敌人
- 行动时优先技能
- 不做目标策略
- 不做长期规划

后续 AI 策略应以战斗命令为边界，不直接修改战场状态。

## 14. 战斗结果投影

战场结束后不应直接写存档。

战场系统应输出 battle result projection：

- 结果状态：胜利、失败、逃跑、中断
- 存活单位
- 死亡单位
- 造成伤害、击杀、治疗等统计
- 掉落或奖励候选
- 需要外部世界处理的事件
- 需要写回角色实例的成长变化

由应用层或地图/事件层决定如何把结果写回：

- 当前队伍
- 地图状态
- 剧情 flag
- 交互对象状态
- 存档记录

## 15. 与地图/剧情系统的关系

游戏不只有战斗。战场系统应作为地图/剧情流程中的一个可进入、可退出的子会话。

建议关系：

- `WorldSession` 或 `MapSession` 持有当前地图与事件进度。
- 地图交互或剧情事件生成 `BattleEncounter`。
- 应用层根据 encounter 创建战场 session。
- 战场 session 完成后输出 battle result projection。
- 地图/剧情层消费 projection 并继续推进。

战场系统不直接知道：

- 哪个宝箱已打开
- 哪段剧情已播放
- 哪个地图对象应消失
- 下一张地图在哪里

这些都属于战斗外系统。

## 16. 暂时移除时应保留的设计边界

如果为了重构暂时移除战场相关代码，建议至少保留本文和以下设计结论：

- 战场 session 是普通运行态对象，不是单例。
- 战场输入是命令，输出是事件。
- 战场运行应 deterministic。
- 战场单位是角色实例的单场投影。
- 存档不保存 runtime definition 引用。
- 战斗动作来自技能/物品/特殊能力等来源的运行时投影。
- attachment source 与 hook 继续统一，但 hook 执行要受固定 phase 约束。
- 地图/剧情通过 encounter 和 result projection 与战场衔接。

不建议保留旧实现作为兼容层。如果要移除，就应移除到清晰边界，再按本文重新实现。

## 17. 重构落地顺序

建议后续按以下顺序重建：

1. 定义 `BattlefieldDefinition`、`BattleEncounterDefinition`、`BattleParticipantDefinition` 的边界。
2. 定义 `BattleSession` / `BattleState` 运行态，不接 UI，不接 Godot。
3. 定义完整命令模型，先补 `UseSkill` 的目标选择。
4. 重建格子、移动、占位、控制区和可达格计算。
5. 重建行动条推进与阶段状态机。
6. 重建战斗动作投影层，把技能学习态转换为动作集合。
7. 重建伤害、治疗、资源、冷却、Buff 结算。
8. 接回 attachment / hook，但先补 phase 合法矩阵和来源归因。
9. 接回结构化事件、确定性回放测试。
10. 最后接地图 encounter 与 result projection。

## 18. 不做的事

战场系统第一版重构不追求：

- 兼容旧 `BattleEngine` 类名和方法签名
- 支持真实渲染或 Godot 节点
- 做成通用战棋引擎
- 支持任意脚本化战斗规则
- 在核心层读取 JSON 文件
- 在核心层直接写存档
- 在战场内处理完整剧情系统

第一版应先把单场战斗的输入、状态、结算、事件和结果边界立稳。
